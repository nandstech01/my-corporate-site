
### メリット

- ✅ **完全無料**（$0/月）
- ✅ **学習効果が高い**（制約の中で最適化を学べる）
- ✅ **シンプルな構成**（1台のVMで完結）

### デメリット

- ❌ **不安定**（OOM再起動が頻発）
- ❌ **パフォーマンスが悪い**（スワップ発生時）
- ❌ **トラブルシューティングに時間がかかる**
- ❌ **スケールしない**（アクセスが増えると即座に限界）

### 向いている用途

- **学習目的**（GCP無料枠の制約を理解したい）
- **短期間のテスト**（1-2週間だけ動かす）
- **トラブルシューティングも学習の一環**として受け入れられる

### 向いていない用途

- **実用的な評価基盤**として使いたい
- **安定稼働**を期待する
- **トラブルシューティングの時間を最小化**したい

### 技術的な問題点

#### 1. OOM（Out of Memory）キラー

```bash
# /var/log/syslog で頻繁に見られるエラー
kernel: Out of memory: Killed process 1234 (python) 
       total-vm:500000kB, anon-rss:400000kB, file-rss:0kB
```

**原因**: 3サービスの合計メモリ使用量が1GBを超える

**影響**: プロセスが強制終了され、サービスが停止

---

#### 2. スワップスラッシング

**現象**: スワップ領域を作っても、CPU性能（0.25 vCPU）で処理が追いつかない

```bash
# スワップ使用量が増加すると応答時間が大幅に悪化
$ free -h
              total        used        free      shared  buff/cache   available
Mem:          1.0Gi       950Mi        50Mi       1.0Mi        50Mi        50Mi
Swap:         2.0Gi       1.5Gi       500Mi  # ← スワップを大量に使用
```

**影響**: API応答時間が10秒以上に増加

---

#### 3. 予測不可能な挙動

**問題**: どのプロセスがOOMキラーで停止するかは予測不可能

- Djangoが停止 → RAG API が使えない
- Grafanaが停止 → ダッシュボードが見れない
- MLflowが停止 → 評価結果の記録ができない

## B案: Django常時 + Grafana/MLflow必要時起動（推奨）⭐

### 構成図

```
┌─────────────────────────────────────────────┐
│   GCP Compute Engine (e2-micro, 1GB RAM)    │
│                                             │
│  Docker Compose                             │
│  ┌──────────────────────────────────────┐  │
│  │  Django Backend :8000 （常時稼働）   │  │
│  │  メモリ: 200-300MB                    │  │
│  └──────────────────────────────────────┘  │
│                                             │
│  ┌──────────────────────────────────────┐  │
│  │  Grafana :3001 （評価時のみ起動）    │  │
│  │  メモリ: 200-300MB                    │  │
│  │  ↑ 必要な時だけ: docker start grafana│  │
│  └──────────────────────────────────────┘  │
│                                             │
│  ┌──────────────────────────────────────┐  │
│  │  MLflow :5000 （評価時のみ起動）     │  │
│  │  メモリ: 150-200MB                    │  │
│  │  ↑ 必要な時だけ: docker start mlflow │  │
│  └──────────────────────────────────────┘  │
│                                             │
│  常時メモリ: 300-450MB（空き: 550-700MB）  │
│  評価時メモリ: 650-950MB（空き: 50-350MB） │
└─────────────────────────────────────────────┘
```

### メリット

- ✅ **完全無料**（$0/月）
- ✅ **安定稼働**（OOMリスクが低い）
- ✅ **無駄なリソース消費がない**（使わない時は停止）
- ✅ **実用的**（評価時だけダッシュボード確認すれば十分）
- ✅ **スケーラブル**（必要に応じてC案へ移行可能）

### デメリット

- ⚠️ **常時ダッシュボード閲覧はできない**（評価時のみ）
- ⚠️ **手動起動が必要**（またはcronで自動化）
- ⚠️ **評価時は一時的に負荷が上がる**（短時間なら許容範囲）

### 向いている用途

- ✅ **自分の評価基盤**として使う
- ✅ **週1-2回、評価を実施する**
- ✅ **評価結果は定期的に確認すれば良い**
- ✅ **完全無料で運用したい**

### 向いていない用途

- ❌ **常時ダッシュボード表示が必要**
- ❌ **複数人が同時にアクセスする**
- ❌ **リアルタイムモニタリングが必要**

### 運用方法

#### 1. 評価開始時

```bash
# SSHでVMにログイン
gcloud compute ssh rag-eval-vm --zone=us-west1-b

# Grafana/MLflowを起動
docker-compose start grafana mlflow

# 起動確認
docker-compose ps
```

#### 2. 評価実施

```bash
# 評価を実行
docker exec rag_backend python /app/rag_api/management/commands/evaluate_final.py baseline v2.0

# Grafanaで結果確認
# ブラウザで http://VM_IP:3001 にアクセス

# MLflowで詳細確認
# ブラウザで http://VM_IP:5000 にアクセス
```

#### 3. 評価終了時

```bash
# Grafana/MLflowを停止
docker-compose stop grafana mlflow

# メモリ確認
free -h
```

#### 4. 自動化（オプション）

```bash
# 週次でGrafana/MLflowを起動するcron
# crontab -e

# 毎週日曜 9:00に起動
0 9 * * 0 cd /home/user/app && docker-compose start grafana mlflow

# 毎週日曜 18:00に停止
0 18 * * 0 cd /home/user/app && docker-compose stop grafana mlflow
```
