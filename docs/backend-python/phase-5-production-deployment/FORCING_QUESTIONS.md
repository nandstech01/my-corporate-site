# Phase 5: フォーシング・クエスチョン（実装前の必須回答）

**最終更新**: 2026-01-03  
**目的**: 無駄な実装を避けるため、Phase 5 の実装前に回答すべき3つの質問

---

## ⚠️ なぜこの質問が重要か？

Phase 5 の最適なアーキテクチャは、**あなたの用途と優先順位**によって大きく変わります。

- 「外部公開」と「自分の評価基盤」では、**必要な可用性が真逆**
- 「常時表示」と「必要時のみ」では、**必要なリソースが2-3倍違う**
- 「完全無料」と「月数ドル許容」では、**安定性が大幅に変わる**

**この3つの質問に答えずに実装すると、無駄な実装・学習・トラブルシューティングに時間を費やします。**

## 📋 3つのフォーシング・クエスチョン

### Question 1: これは「外部公開」用途ですか？それとも「自分の評価基盤」用途ですか？

#### なぜこの質問が重要か？

- **外部公開（複数人が使う）**: 
  - 常時稼働が必須
  - 高可用性（99%以上）が求められる
  - ネットワーク転送量が多い（複数人がアクセス）
  - セキュリティ要件が高い

- **自分の評価基盤（1人で使う）**:
  - 使うときだけ起動で十分
  - 多少の停止時間は許容できる
  - ネットワーク転送量は少ない
  - セキュリティは最低限でOK

#### 回答の影響

| 回答 | 推奨アーキテクチャ | 理由 |
|------|------------------|------|
| **外部公開** | C案（役割分離、有料） | 安定性と可用性が必要 |
| **自分の評価基盤** | B案（Django常時、他は必要時） | 無料枠で十分、無駄がない |

### Question 2: Grafana/MLflowは**常時見る必要**がありますか？それとも「評価するときだけ」で足りますか？

#### なぜこの質問が重要か？

- **常時表示が必要**:
  - Grafana/MLflowを常時起動 → メモリ 650-950MB消費
  - e2-micro（1GB）では**OOM（Out of Memory）のリスクが高い**
  - 有料プラン（e2-small: 2GB, 月$13程度）が必要

- **評価時だけで足りる**:
  - Django常時 + Grafana/MLflow必要時起動 → メモリ 300-450MB
  - e2-micro（1GB）でも**十分な余裕**（550-700MB空き）
  - 無料枠で安定稼働

#### 回答の影響

| 回答 | 推奨アーキテクチャ | メモリ使用量 | コスト |
|------|------------------|------------|-------|
| **常時表示が必要** | C案（役割分離、有料） | 役割分離で安定 | 月$10-20 |
| **評価時だけで足りる** | B案（必要時起動） | 300-450MB（常時） | 無料 |

#### 典型的な使用パターン

**例1: 常時表示が必要なケース**
- 毎日、複数回ダッシュボードを確認する
- リアルタイムでメトリクスを監視したい
- チームメンバーがいつでもアクセスする

**例2: 評価時だけで足りるケース**
- 週に1-2回、評価を実施する
- 評価結果を確認するだけ（リアルタイムモニタリング不要）
- 自分1人で使う

### Question 3: "無料に固執して不安定"と "月数ドルで安定"なら、どちらがあなたのKPI（時間・信用・学習効率）に効きますか？

#### なぜこの質問が重要か？

Phase 5 の選択肢は、**コストと安定性のトレードオフ**です：

- **無料に固執**:
  - メリット: 完全無料（$0/月）
  - デメリット: 不安定（OOM再起動、転送量制限）、トラブルシューティングに時間がかかる

- **月数ドル許容**:
  - メリット: 安定稼働、トラブルが少ない、学習に集中できる
  - デメリット: 月$10-20のコスト

#### トレードオフの具体例

| 項目 | 無料枠（e2-micro、不安定） | 有料（e2-small等、安定） |
|------|-------------------------|----------------------|
| **月額コスト** | $0 | $10-20 |
| **メモリ** | 1GB（不足しやすい） | 2GB以上（十分） |
| **安定性** | ⚠️ OOM再起動のリスク | ✅ 安定 |
| **トラブルシューティング** | ⚠️ 頻繁に必要 | ✅ ほとんど不要 |
| **学習効率** | ⚠️ インフラに時間を取られる | ✅ アプリケーション開発に集中 |

#### あなたのKPIは何ですか？

**例1: 時間を最優先（学習効率）**
- 「トラブルシューティングに時間をかけたくない」
- 「アプリケーション開発に集中したい」
- → **有料プラン推奨**（月$10-20で安定）

**例2: コストを最優先（完全無料）**
- 「GCP無料枠の制約を学びたい」
- 「トラブルシューティングも学習の一環」
- → **無料枠推奨**（不安定だが$0）

**例3: バランス（実用性）**
- 「自分の評価基盤として実用的に使いたい」
- 「評価時だけGrafana/MLflowを起動すれば十分」
- → **B案推奨**（無料枠で安定）

## 🎯 回答結果とアーキテクチャのマッピング

### パターン1: 外部公開 + 常時表示 + 安定優先

**推奨**: **C案（役割分離、有料）**

- Django: e2-micro（無料枠）で常時稼働
- Grafana/MLflow: e2-small（有料、月$13）で常時稼働
- 月額: $13-20
- 安定性: ★★★★★

---

### パターン2: 自分の評価基盤 + 評価時のみ + 完全無料

**推奨**: **B案（Django常時、他は必要時起動）**

- Django: 常時稼働（RAG API提供）
- Grafana/MLflow: 評価時のみ手動起動
- 月額: $0（無料枠内）
- 安定性: ★★★★☆

---

### パターン3: 自分の評価基盤 + 常時表示 + 完全無料

**推奨**: **A案（全部載せ、非推奨）**

- Django + Grafana + MLflow を e2-micro で常時稼働
- 月額: $0（無料枠内）
- 安定性: ★☆☆☆☆（OOM再起動頻発）

**警告**: このパターンは**非推奨**です。以下のいずれかを妥協してください：
- 「常時表示」を諦める → **B案**へ
- 「完全無料」を諦める → **C案**へ

---

### パターン4: 外部公開 + 評価時のみ + 完全無料

**推奨**: **B案（Django常時、他は必要時起動）**

- Django: 常時稼働（RAG API提供）
- Grafana/MLflow: cron等で定期起動、または手動起動
- 月額: $0（無料枠内）
- 安定性: ★★★☆☆（ただし外部公開には不向き）

**注意**: 外部公開の場合、可用性が求められるため、本来は**C案**が推奨です。
