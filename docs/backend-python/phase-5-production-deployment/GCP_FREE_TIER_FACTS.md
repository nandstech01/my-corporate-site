## 🎯 GCP無料枠の2つのタイプ

GCPの無料枠には**2つのタイプ**があります：

### 1. Always Free（常時無料）

- **期限**: なし（永続的に無料）
- **条件**: 特定のサービス・リージョン・使用量制限内
- **対象**: e2-microインスタンス、標準永続ディスク30GB、など

### 2. 12ヶ月無料トライアル

- **期限**: 新規アカウント作成から12ヶ月間
- **クレジット**: $300分のクレジット
- **対象**: ほぼ全てのGCPサービス

**Phase 5では "Always Free" を活用します。**

## ❌ よくある誤解と正しい理解

### 誤解1: Cloudflareで GCP の転送量課金を回避できる

#### ❌ 誤った理解

「Cloudflareの無料プランを使えば、GCPの転送量制限（1GB/月）を回避できる」

#### ✅ 正しい理解

**GCP VMからのアウトバウンド通信は必ずGCPの転送量対象**です。

**理由**:
1. Cloudflareは "ユーザー→Cloudflare" の転送を無料にする
2. しかし、"GCP VM→Cloudflare→ユーザー" の流れでは、**"GCP VM→Cloudflare"の部分はGCPの転送量**
3. つまり、Cloudflareを使っても**GCPの1GB/月制限は回避できない**

#### 図解

```
❌ 誤った理解（Cloudflareを使えばGCPの転送量が無料）
┌──────────┐         ┌─────────────┐         ┌──────────┐
│  GCP VM  │ ─────→ │ Cloudflare  │ ─────→ │  ユーザー  │
└──────────┘         └─────────────┘         └──────────┘
                      ↑ここが無料だから全部無料？

✅ 正しい理解（GCP→Cloudflareは必ずGCP課金対象）
┌──────────┐         ┌─────────────┐         ┌──────────┐
│  GCP VM  │ ─────→ │ Cloudflare  │ ─────→ │  ユーザー  │
└──────────┘         └─────────────┘         └──────────┘
    ↑                       ↑
  GCP課金対象           Cloudflare無料
  （1GB/月制限）         （無制限）
```

#### Cloudflareの正しい活用方法

Cloudflareは**静的アセットのキャッシュ**で転送量を削減できます：

```
1回目のリクエスト:
ユーザー → Cloudflare → GCP VM（画像取得）→ Cloudflare（キャッシュ）→ ユーザー
                         ↑ GCP転送量消費

2回目以降のリクエスト:
ユーザー → Cloudflare（キャッシュから返す）→ ユーザー
           ↑ GCP転送量消費なし（Cloudflareが代わりに返す）
```

**効果**: 静的アセット（画像、CSS、JS）の転送量を削減できる（動的コンテンツは効果なし）

### 誤解2: e2-micro（1GB）で3サービス常時稼働が実用的

#### ❌ 誤った理解

「Django + Grafana + MLflow を e2-micro（1GB RAM）で常時稼働させる」

#### ✅ 正しい理解

**1GBメモリで3サービス常時稼働は極めて不安定**です。

#### メモリ使用量（実測値の目安）

| サービス | アイドル時 | ピーク時 | 備考 |
|---------|-----------|---------|------|
| **Django** | 150-200MB | 250-300MB | Gunicorn 2 workers |
| **Grafana** | 150-200MB | 250-300MB | ダッシュボード表示時 |
| **MLflow** | 100-150MB | 150-200MB | SQLite使用 |
| **システム** | 100-150MB | 100-150MB | OS、Docker等 |
| **合計** | **500-700MB** | **750-950MB** | - |
| **空きメモリ** | **300-500MB** | **50-250MB** ⚠️ | OOMリスク高 |

**問題点**:
1. ピーク時にメモリが不足し、**OOM（Out of Memory）キラー**が発動
2. プロセスが強制終了され、サービスが停止
3. スワップを使っても、CPU性能（0.25 vCPU）で処理が追いつかない

#### 現実的な構成

**選択肢B**: Django常時 + Grafana/MLflow必要時起動

| サービス | 常時稼働時 | 評価時（全起動） |
|---------|-----------|----------------|
| Django Backend | 200-300MB | 200-300MB |
| Grafana | - | 200-300MB |
| MLflow | - | 150-200MB |
| システム/その他 | 100-150MB | 100-150MB |
| **合計** | **300-450MB** | **650-950MB** |
| **空きメモリ** | **550-700MB** ✅ | **50-350MB** ⚠️ |

**評価時は一時的に負荷が上がるが、短時間（数分〜数十分）なら許容範囲**

### 誤解3: 静的IPは必要に応じて$3/月

#### ⚠️ 部分的に正しいが補足が必要

**正確な情報**:

| 状況 | 課金 | 金額 |
|------|------|------|
| **使用中のVMに紐づく静的IP** | 無料 | $0 |
| **停止中のVMに紐づく静的IP** | 課金対象 | $3/月程度 |
| **未使用の予約IP** | 課金対象 | $3/月程度 |

**推奨**: VMを常時稼働させるなら、静的IPは無料で使える ✅

**参考**: [Stack Overflow - GCE Static IP Pricing](https://stackoverflow.com/questions/49501329/google-compute-engine-assigning-static-ip-pricing)

## 📏 Phase 5 での現実的な制約

### 1. メモリ制約（最大の制約）

**e2-micro: 1GB RAM**

- Django常時稼働: 200-300MB
- Grafana/MLflow常時稼働: 350-500MB
- システム: 100-150MB
- **合計: 650-950MB**（空き: 50-350MB ⚠️）

**対策**:
- ✅ **選択肢B**: Django常時 + Grafana/MLflow必要時起動
- ❌ 選択肢A: 3サービス常時稼働（OOMリスク高）
- ✅ 選択肢C: 役割分離（有料だが安定）

---

### 2. ネットワーク転送制約

**北米からのアウトバウンド: 1GB/月**

#### 1GB/月の目安

| 用途 | 1回あたり | 月間回数（1GB上限） |
|------|-----------|------------------|
| Grafanaダッシュボード表示 | 500KB-1MB | 1,000-2,000回 |
| MLflow UI表示 | 300KB-500KB | 2,000-3,300回 |
| RAG API（JSON応答） | 5-10KB | 100,000-200,000回 |

**現実的な使用パターン**:
- **自分1人で評価**: 月10-20回ダッシュボード表示 → 10-20MB（**余裕あり** ✅）
- **常時表示（毎日確認）**: 月100回以上 → 50-100MB（**余裕あり** ✅）
- **外部公開（複数人）**: 月1,000回以上 → 500MB-1GB（**上限ギリギリ** ⚠️）

**対策**:
- ✅ Cloudflareで静的アセットをキャッシュ（画像、CSS、JS）
- ✅ Grafanaの画像埋め込みを最小化
- ⚠️ 外部公開の場合は転送量を監視

---

### 3. CPU制約

**e2-micro: 0.25 vCPU（バースト可能）**

- **通常時**: 0.25 vCPUで十分（DjangoのAPI応答）
- **評価実行時**: 短時間バーストで対応可能
- **問題**: スワップ処理時にCPUが追いつかない（メモリ不足時）

**対策**:
- ✅ メモリ不足を避ける（選択肢B推奨）

---

### 4. ディスク制約

**標準永続ディスク: 30GB**

#### ディスク使用量の目安

| 項目 | 容量 |
|------|------|
| OS（Ubuntu 22.04） | 4-5GB |
| Docker images | 2-3GB |
| MLflow artifacts（評価結果） | 1-2GB（100 runs） |
| Grafana data | 500MB-1GB |
| ログ | 1-2GB |
| **合計** | **9-13GB** |
| **空きディスク** | **17-21GB** ✅ |

**対策**:
- ✅ 古いMLflow runを定期削除（90日以上）
- ✅ Dockerイメージの定期クリーンアップ
- ✅ ログローテーション設定

## 🎯 結論

### Phase 5 で推奨するのは「選択肢B」

**理由**:
1. ✅ **無料枠で安定稼働**（OOMリスクが低い）
2. ✅ **転送量も余裕**（月50-100MBで済む）
3. ✅ **無駄なリソース消費がない**
4. ✅ **実用的**（評価時だけGrafana/MLflowを起動）

**向いている用途**:
- 自分の評価基盤として使う
- 週1-2回、評価を実施する
- 完全無料で運用したい
