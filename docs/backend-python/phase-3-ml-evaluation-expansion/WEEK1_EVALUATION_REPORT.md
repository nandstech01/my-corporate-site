## 🔍 3. 定性分析

### 3.1 BM25の成功要因

1. **固有名詞の強み**:
   - BM25（全文検索）は固有名詞を正確に検出
   - 例: "Mike King" → `faq-2`, "SAP" → `case-2` を正確に検索

2. **Vector検索との相補性**:
   - Vector検索: 意味的に類似したコンテンツを検索
   - BM25: 単語レベルでの完全一致を検索
   - 両者を組み合わせることで、精度が向上

3. **実装の安定性**:
   - PostgreSQL の `ts_rank_cd` を使用したBM25実装は安定している
   - `content_tsvector` カラムが正しく全文検索インデックスとして機能

### 3.2 RRFの失敗要因

1. **パラメータの不適切さ**:
   - `k=60` がデフォルト値だが、最適ではない可能性
   - `vector_weight=1.0`, `bm25_weight=1.0` のバランスが不適切

2. **統合ロジックの課題**:
   - RRFはVector検索とBM25検索の「両方に出現するもの」を優遇
   - しかし、今回のケースでは「片方だけに出現するもの」が正解の場合が多い
   - 例: 固有名詞は BM25 で強く検出されるが、Vector検索では弱い
   - RRF統合により、こうした「片方の強み」が薄まった可能性

3. **Fragment IDの質の問題**:
   - 正解セット（`expected_fragment_ids`）が不完全な可能性
   - RRFで新たに上位に来たFragment IDが「正解ではない」と判定されている可能性

## 💡 4. 具体的な改善提案

### 4.1 短期（Phase 3 Week 2）

#### 提案1: BM25をメイン手法として採用
- **理由**: 既にbaseline比で改善効果を確認
- **実装**: `use_bm25=True` をデフォルトに設定
- **期待効果**: Precision@5 で +1.60%, MRR で +8.04% 改善

#### 提案2: 評価データセットの精査
- **課題**: 現在の正解セット（`expected_fragment_ids`）が不完全な可能性
- **対応**:
  1. 各クエリに対して、実際の検索結果（top 20）を人間がレビュー
  2. 「正解だが expected_fragment_ids に含まれていない」ものを追加
  3. 逆に、「不正解だが expected_fragment_ids に含まれている」ものを削除
- **期待効果**: 評価の精度が向上し、真の改善効果を測定可能

#### 提案3: カテゴリ別評価
- **目的**: 固有名詞と一般語で効果が異なることを定量化
- **実装**: `EvaluationService` にカテゴリ別集計機能を追加済み
- **期待効果**: 「どのクエリ種別で改善しているか」を明確化

### 4.2 中期（Phase 3 Week 3〜4）

#### 提案4: RRFパラメータの最適化
- **目的**: RRFの潜在能力を引き出す
- **実装**:
  1. k値を変化させて評価（k=10, 30, 60, 100）
  2. 重みを変化させて評価（vector:bm25 = 1:1, 1:2, 2:1, 1:3）
  3. グリッドサーチで最適パラメータを探索
- **期待効果**: RRFがBM25を上回る可能性

#### 提案5: Recall@k, NDCG の実装
- **目的**: Precision@5, MRR だけでなく、多角的に評価
- **実装**: `EvaluationService` に Recall@k, NDCG を追加
- **期待効果**: 評価の信頼性が向上
