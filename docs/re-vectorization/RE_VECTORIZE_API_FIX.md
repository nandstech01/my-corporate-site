# 再ベクトル化API修正レポート

## 📋 問題の発見

### 発見日時
2025-12-14

### 問題の内容
初回テストで**エラー数39個**が発生。原因は既存のFragment Vectorが削除されずに残っており、重複エラーが発生していた。

### エラーログ
```
エラー数: 39個
重複: Fragment ID xxx は既に存在します
重複: Fragment ID yyy は既に存在します
...（39回繰り返し）
```

---

## 🔍 根本原因分析

### 既存API (`/api/vectorize-blog-fragments`) の問題

**設計目的**: 新規記事のFragment ID初回ベクトル化

**処理フロー**:
```
1. 記事取得
2. Fragment ID抽出
3. ベクトル化
4. 保存（重複チェックあり） ← ここで止まる
```

**問題点**:
- ✅ 新規記事には問題なし
- ❌ **再ベクトル化には不適切**（既存ベクトルを削除しない）
- ❌ 重複チェックで「既に存在します」エラー
- ❌ 結果：古いベクトルがそのまま残る

---

## 🛡️ 安全な解決策

### 選択した方法：新規API作成

**理由**:
1. **既存APIは変更しない**
   - `/api/vectorize-blog-fragments` は記事生成時に使用
   - 変更すると既存の記事生成フローに影響
   - **影響範囲**: 記事生成、ハイブリッド記事生成等

2. **完全に独立した処理**
   - 新規API `/api/re-vectorize-post` を作成
   - 編集ページ専用
   - 既存システムに一切影響なし

---

## 🚀 新規API仕様

### エンドポイント
`POST /api/re-vectorize-post`

### リクエスト
```json
{
  "postId": 121
}
```

### 処理フロー
```
1. 記事情報取得（posts テーブル）
   ↓
2. page_path構築（/posts/{slug}）
   ↓
3. 🗑️ 既存Fragment Vector削除（page_pathで厳密にフィルタ）
   ↓
4. Fragment ID再抽出（マークダウン解析）
   ↓
5. OpenAI Embeddings再生成
   ↓
6. fragment_vectors テーブル保存
   ↓
7. 結果返却
```

### レスポンス
```json
{
  "success": true,
  "message": "記事のFragment ID再ベクトル化完了",
  "result": {
    "postId": 121,
    "postTitle": "記事タイトル",
    "slug": "article-slug",
    "pagePath": "/posts/article-slug",
    "deletedVectors": 39,
    "fragmentsFound": 15,
    "vectorizedCount": 15,
    "successRate": "100.0%",
    "errors": []
  }
}
```
